## Why

力量训练者在记录训练时的核心痛点是：**缺乏一个能在组间休息时以最小心智负担记录数据，同时能在中长期严格执行“容量/强度自调节（Auto-regulation）”逻辑的App。**

当下市面上的绝大部分App仅支持“静态记录本”打卡（例如：计划规定做多少就显示多少），无法执行像 GZCLP, 5/3/1 等含有复杂进度自调解（比如：失败则退跑特定容量，成功则加一定重量）的系统计划。

我们需要在多端 (iOS, Android, macOS) 构建一个极致本地化的应用。利用 Flutter 实现跨平台极简 UI，通过纯计算节点式的规则引擎和本地无模式数据库 (NoSQL) 实现无需后端的“零延迟数据流转”与极其硬核的计划自调解能力。通过基于 JSON 转储机制解决计划的分享和备份问题。

## What Changes

本变更是构建 Fittin v2 核心逻辑的基础，主要包含以下内容：
1. **纯代码实现的训练规则引擎 (Dart)**：引擎能够解析用户导入的 JSON 格式“计划蓝图”，并在每次用户完成训练时，自动根据“是否成功”、“失败的组数”等条件计算下一个循环的训练容量(Sets/Reps)和强度(Weight)。
2. **Local-First 数据架构模型**：引入 Isar (或等价的高性能本地 DB)，定义 App 中的 Template (静态模板) 与 Instance (正在运行的计划实例) 两套数据结构，分离计划规定与用户打卡记录。
3. **极简交互范式 (UI Layer)**：不再强制要求每次组间手动输入重量和次数；默认提供引擎推算出的数值进行 "Zero-Type (零输入打卡)"，并结合单手友好的微调功能和底层休息计时器。
4. **去中心化的 JSON 计划导入导出机制**：放弃繁重的云端模板社区，使用 Base64 URL Scheme 或 二维码形式在端侧直接完成计划库的传输。

## Capabilities

### New Capabilities

- `plan-rule-engine`: 定义了如何从 JSON 中解析复杂的组次数规则与进退阶（Progression & Regression）触发器。
- `local-datastore-schema`: 本地数据库 schema 设计，包括分离的静态模板集和用户的当前实例进度。
- `zero-typing-ui`: 训练中页面的交互方式定义（含默认数据填充、平滑微调手势、跨动作全局休息计时）。
- `peer-to-peer-sharing`: 基于 JSON（可通过二维码或链接）导出和导入模板数据的序列化与反序列化系统。

## Impact

- **架构选型**：彻底抛弃云 BaaS (如 CloudBase)，转移至单机纯本地应用架构。
- **技术栈**：要求选用高并发/低延迟本地库 (Isar) 作为持久层，采用无额外开销的 Dart PODOs (纯对象) 用于规则推演，选用 Flutter Riverpod / Provider 处理激烈的 UI 组内变化。
- **离线能力**：实现 100% Offline 随时可用，无网络时一切逻辑能完全闭环结算和推演下一步重量。
